# systemd-resolved DNS Triage

## When to Use

DNS resolution fails on a Linux host:

- `ping <hostname>` returns "名称或服务未知" / "Temporary failure in name resolution"
- Tailscale MagicDNS (`*.ts.net`) stops resolving even though `tailscale dns status` shows MagicDNS enabled
- `resolvectl status` times out or produces no output
- Short hostnames via Tailscale search domains stop working
- Hostnames work from some machines in the same network but not others

## Diagnostic Flow

### Step 1: Check resolv.conf mode

Two modes determine how DNS queries are processed:

```bash
cat /etc/resolv.conf
ls -la /etc/resolv.conf
```

**Stub mode** (`/etc/resolv.conf` → `127.0.0.53`): systemd-resolved stub resolver handles all DNS. Search domain expansion is done by resolved. `resolvectl` is the primary diagnostic tool.

**Foreign mode** (`/etc/resolv.conf` managed directly — e.g. `# generated by tailscale`, `nameserver 100.100.100.100`): glibc resolver handles DNS directly via the listed nameserver. `resolvectl` may still work if systemd-resolved is running, but `/etc/resolv.conf` is the authoritative config. Search domain expansion follows glibc rules (`ndots` setting).

### Step 2: Check the upstream (resolved's real config)

```bash
cat /run/systemd/resolve/resolv.conf
```

Only relevant in **stub mode**. **Key signal: "No DNS servers known"** — systemd-resolved has no upstream DNS servers configured. DNS queries to 127.0.0.53 have nowhere to forward.

In **foreign mode**, check the actual `/etc/resolv.conf` instead — this IS the upstream.

### Step 3: Check D-Bus interface health

```bash
timeout 5 resolvectl status
```

Only relevant in **stub mode**. **Key signal: command times out** — the D-Bus service `org.freedesktop.resolve1` is unresponsive. Note that `systemctl status systemd-resolved` will still show `Active: active (running)` — D-Bus can hang independently of the daemon process.

### Step 4: Check the actual service state

```bash
systemctl status systemd-resolved
```

Even if D-Bus is hung, the service will show `active (running)`. Don't be fooled — this is not a reliable indicator of D-Bus health. In foreign mode, systemd-resolved may still be running but queries bypass it entirely.

### Step 5: Check for port 53 conflicts

Multiple DNS listeners on the same host can interfere:

```bash
ss -tlnp 2>/dev/null | grep ':53 '
ss -ulnp 2>/dev/null | grep ':53 '
```

Expected: `127.0.0.53:53` (systemd-resolved stub). Anything else on `:53` (e.g. `127.0.0.54:53`, another `0.0.0.0:53`) indicates a competing DNS service (Docker, dnsmasq, another VPN client) that may intercept or conflict with queries.

Also check for unexpected listeners on non-standard ports:

```bash
ss -tlnp 2>/dev/null | grep -E ':(53|[0-9]*[0-9]{3,})$' | grep -v '5353\|5355'
```

### Step 6: Check what Tailscale thinks

```bash
tailscale dns status
```

**Key signal:** shows "Tailscale DNS: enabled" + "MagicDNS: enabled" + correct resolvers/search domains — **but systemd-resolved never received them** because D-Bus calls silently failed. Tailscale doesn't validate D-Bus return codes, so it reports success when the config was never applied.

Also check how Tailscale manages the config:

```bash
tailscale debug restun 2>&1 | head -5 || true
```

### Step 7: Check search domains

```bash
grep search /etc/resolv.conf
```

**Key signal (stub mode):** `search .` (single dot) instead of expected domains like `tail2e6efb.ts.net`. This confirms systemd-resolved has no configured search domains.

**Key signal (foreign mode):** search domains may look correct in `/etc/resolv.conf` but still fail to expand for certain name patterns — see "Search Domain Expansion" below.

### Step 8: Cross-machine comparison

When the same hostname works on one machine but not another, compare:

```bash
# DNS resolver chain (stub vs foreign)
cat /etc/resolv.conf

# Search domains
grep search /etc/resolv.conf

# Underlying resolution
getent hosts <hostname>
dig <hostname> +short
resolvectl query <hostname> 2>&1

# Actual connectivity
curl -svo /dev/null 'http://<hostname>/' 2>&1 | head -5
```

## Root Cause

systemd-resolved's D-Bus interface (`org.freedesktop.resolve1`) hangs while the daemon process itself stays running. This can happen after:

- Long system uptime / accumulated D-Bus message queue pressure
- Multiple rapid DNS configuration changes (Docker, VPN, Tailscale)
- System suspend/resume cycles where D-Bus reconnection doesn't complete
- A transient D-Bus bus daemon stutter

When D-Bus is hung, `resolvectl` (which talks to resolved exclusively over D-Bus) cannot communicate. Tailscale's `tailscaled` also injects DNS config via D-Bus — the calls appear to succeed but systemd-resolved never processes them.

## Search Domain Expansion

Search domain expansion differs between stub and foreign resolv.conf modes — this is a common source of confusion.

### Stub mode (systemd-resolved handles /etc/resolv.conf)

Search domains are configured per-link or globally via systemd-resolved. Expansion is reliable for all name patterns:

```bash
# In stub mode, systemd-resolved handles expansion
host searxng.home          # → tries searxng.home.<search1>, searxng.home.<search2>, ...
host searxng.serverhome    # → tries searxng.serverhome.<search1>, ...
```

All patterns get expanded. Works correctly across all search domains.

### Foreign mode (Tailscale or other agent writes /etc/resolv.conf directly)

```bash
# resolv.conf starts with:
# # resolv.conf(5) file generated by tailscale
# nameserver 100.100.100.100
# search tail2e6efb.ts.net home.z-core.cn ...
```

glibc resolver handles expansion, and **glibc's `ndots` setting matters**:

| Name pattern | ndots=1 (default) | ndots=0 |
|---|---|---|
| `serverhome` (0 dots) | ✅ search domain expanded | ✅ search domain expanded |
| `searxng.home` (1 dot) | ❌ tried as-is first (NXDOMAIN), then short-circuit | ✅ search domain expanded |
| `searxng.home.z-core.cn` (2 dots) | ✅ search domain expanded | ✅ search domain expanded |

**Rule of thumb:** In foreign mode, short single-word hostnames (like `serverhome`) expand fine. Names with dots (like `searxng.home`) may fail because glibc tries them as absolute FQDNs first and the NXDOMAIN response interferes with search domain retries. Use the FQDN form (`searxng.home.home.z-core.cn`) to bypass expansion issues.

### How to detect the mode

```bash
# Check resolv.conf mode
resolvectl status | grep 'resolv.conf mode'
# → "stub" or "foreign"

# Check if Tailscale manages it
head -1 /etc/resolv.conf
# → "# resolv.conf(5) file generated by tailscale" means foreign mode
```

### When both modes exist on the same tailnet

Different machines in the same tailnet may have different resolv.conf modes even with identical Tailscale config. This is determined by how the OS resolves DNS — some use systemd-resolved stub, others use the raw DNS servers. Cross-machine DNS comparisons must account for this.

### Testing the actual resolution chain

```bash
# Try different name forms to find what works:
getent hosts serverhome                    # search domain expanded form
getent hosts searxng.home                  # may fail on foreign mode
getent hosts searxng.home.home.z-core.cn   # FQDN form — always works
dig searxng.home +short                    # bypasses glibc expansion
resolvectl query searxng.home 2>&1         # only works in stub mode

# Test actual HTTP reachability (not just DNS):
curl -svo /dev/null 'http://<hostname>/' 2>&1 | head -5
curl -s 'http://<hostname>/search?q=test&format=json' | head -c 200
```

### Reverse proxy routing through DNS

DNS resolution is only half the story. When a hostname resolves to a reverse proxy (nginx), the **`Host` header** determines which virtual host serves the request:

```bash
# DNS resolves to the same IP, but nginx routes differently:
curl http://searxng.home/                  # → Host: searxng.home  → SearXNG backend
curl http://search.home/                   # → Host: search.home   → different backend
curl http://searxng.serverhome/            # → Host: searxng.serverhome → yet another backend

# The Host header is derived from the URL, not from DNS resolution.
# Two hostnames that resolve to the same IP can hit completely different services.
```

When debugging "this hostname works from machine A but not B":
1. Check DNS resolution: does B resolve it at all?
2. If yes, check the Host header nginx receives (compare `curl -v`)
3. If no, check resolv.conf mode and search domain expansion rules

## Fix Procedure

### 1. Restart systemd-resolved

```bash
sudo systemctl restart systemd-resolved
sleep 2
```

This resets the D-Bus listener. The daemon re-reads its configuration from uplink DNS servers on connected interfaces.

### 2. Re-inject Tailscale DNS configuration

```bash
tailscale set --accept-dns=false
sleep 1
tailscale set --accept-dns=true
sleep 2
```

Toggling `accept-dns` forces tailscaled to re-issue all D-Bus DNS configuration calls. Without this step, the Tailscale MagicDNS config won't be reapplied.

### 3. Verify

```bash
# D-Bus interface alive?
timeout 5 resolvectl status
# Should show Global DNS Servers: 100.100.100.100 (Tailscale MagicDNS) + local router

# Upstream DNS populated?
cat /run/systemd/resolve/resolv.conf
# Should show nameserver 100.100.100.100 + search domains like tail2e6efb.ts.net

# Magic DNS works?
timeout 5 resolvectl query <peer>.tail2e6efb.ts.net
# Should return the Tailscale IP in milliseconds

# Short names via search domain?
ping -c 1 -W 2 <peer>
# Should expand via search domain and return

# Regular internet DNS still works?
ping -c 1 -W 2 google.com
```

## Pitfalls

- **Do NOT skip Step 2.** Restarting systemd-resolved alone clears all D-Bus DNS config. Tailscale won't automatically re-inject — you must toggle `--accept-dns`.
- **`systemctl status systemd-resolved` showing "active (running)" is NOT proof of DNS health.** Always check `resolvectl status` or `/run/systemd/resolve/resolv.conf` for the real state.
- **Don't confuse `stub-resolv.conf` with `resolv.conf`.** The file at `/run/systemd/resolve/resolv.conf` is the upstream (real) config; `/etc/resolv.conf` (→ `stub-resolv.conf`) is the local stub. "No DNS servers known" only appears in the former.
- **`journalctl -u systemd-resolved` may return no entries** if the user lacks journal read permissions (not in `adm`/`systemd-journal` group). Run with `sudo journalctl -u systemd-resolved` instead.
- **Don't fall back to hardcoded IPs** for Tailscale peers when Magic DNS is down — after fixing DNS the hostnames will work. Using IPs as a workaround means you won't notice when DNS is restored.
- **Keep `resolv.conf mode` in mind.** The diagnostic steps differ between stub and foreign mode (see "Search Domain Expansion" above). In foreign mode, `resolvectl status` may work but is not the authoritative view.
- **Port 53 conflicts are transient.** A competing DNS listener (e.g. `127.0.0.54:53`) may disappear on recheck. The fact it existed at all indicates a daemon restart lifecycle issue (Docker, VPN, or a short-lived process). Document it when seen but don't block on its absence.
- **glibc search domain expansion is not equivalent to systemd-resolved.** Names with one dot (like `searxng.home`) that work on machine A with stub mode will NOT work on machine B with foreign mode. Always test both `getent hosts` AND `curl -sv` when debugging cross-machine DNS.
- **When testing reverse proxy routing, don't trust `getent hosts` alone.** DNS resolution confirms the IP, but the Host header determines the backend. Always `curl -v` to see which HTTP response you actually get.
